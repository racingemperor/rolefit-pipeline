# 简历格式数据库

这个目录保存 ResumeFormatGate 和 ResumeArchitect 共享的静态格式规则。

它不是固定简历模板库，而是“板块规则 + 岗位方向 overlay + HR 信任评分 + 多版本选择 + 驳回条件”的数据库。后续用户端只需要传入候选人材料摘要、目标岗位族、命中的规则 ID 和少量证据引用，不需要反复消耗 token 解释简历应该怎么写。

## 文件

- `resume_format_database.zh-CN.json`：主数据库，包含基础五板块、可选板块、岗位方向 overlay、默认校招通用版、多版本简历、HR 信任评分和校验规则。
- `resume_format_schema.json`：主数据库 schema。
- `summary.json`：规模与版本摘要。

## 基础五板块

用户给定的基础格式被保留为必需板块：

- 学校信息。
- 个人联系方式。
- 掌握技能。
- 项目竞赛经历。
- 个人性格和潜力。

其中“个人性格和潜力”不能写成空泛自评，而应转成证据化表达，例如学习能力、owner 意识、协作、复盘、问题解决，并绑定项目或经历证据。

## 运行方式

推荐流程：

1. `ProfileExtractor` 抽取候选人事实和证据。
2. `ResumeFormatGate` 读取本库，判断材料是否达到生成门槛。
3. 如果 `format_gate_status = pass`，`ResumeArchitect` 选择版本并生成简历草稿。
4. `FactualReviewer` 审查事实、夸大、隐私、NDA 和面试可防御性。

如果用户没有明确目标公司或具体 JD，默认使用 `campus_general_cn_one_page`，即校招通用中文版一页，强调覆盖面广、工程底座和方向证据。

## 多版本策略

当前版本包含 10 个简历版本：

- 校招通用中文版一页。
- 软件/AI 定向中文版。
- 硬件/嵌入式/IC 定向中文版。
- 新能源/车辆/电池定向中文版。
- 科研/算法/研发版。
- 实习短版。
- ATS 纯文本版。
- 校招门户结构化填报版。
- 英文 Resume 版。
- 面试深挖附录。

系统一次最多建议两个版本：一个主投递版，一个探索版或系统填报版，避免让用户维护过多简历。

## 驳回原则

低于 HR 信任评分 60 分，不进入简历生成，只输出补材料清单。

硬驳回条件包括：

- 基础五板块无法解析。
- 缺学校/专业/学历/毕业时间等校招资格信息。
- 无邮箱或电话等可联系渠道。
- 无任何项目、竞赛、实践或可证明经历。
- 技能全是关键词，无法关联项目证据。
- 把参与写成主导，或把未完成写成上线、量产、发表。
- 出现身份证、详细住址、私聊截图、内部数据等隐私或合规风险。

本库只定义格式和门槛，不允许生成虚假经历。
